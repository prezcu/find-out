● Yes, that's exactly the cause. Here's what happens under the hood:

  The flow today
  1. SearchScreen calls repository.searchLocationsByName(...), gets
  List<LocationEntity> back from the network, and renders the cards. Nothing is 
  inserted into Room.
  2. You tap a card with id = X. Navigation pushes attraction/X.
  3. AttractionScreenViewModel calls repository.getLocationById(X) →
  dao.getLocationById(X) → Room runs SELECT * FROM location WHERE id = :id.
  4. Row X isn't in the location table (only the nearby refresh ever wrote rows
  into it, and it clearLocations() first). The Flow<LocationEntity?> emits null.
  5. StateFlow initial value is null, current value is null, screen stays in its
   empty / loading state forever.

  The architecture you have is "Room is the single source of truth, network
  refreshes Room." The search path violates that contract — it bypasses Room
  entirely, so the detail screen (which trusts Room) has nothing to read.

  The fix — make the repository write search results into Room before returning
  them. LocationDao.insertLocations is already @Upsert, so it merges instead of
  wiping:

  libraries/dependencies or what else would you enumerate? what counts as part

● Good question — they're often confused. Here's the mental model and what your
  project specifically has.

  The distinction

  Architecture = the shape of the code: where state lives, how data flows, what
  each layer is allowed to know about. Swap Retrofit for Ktor and the
  architecture is unchanged. Switch from MVVM to MVI and it is changed.

  Tech stack = the concrete tools that fill those slots. Swappable in principle,
   even if rewriting would be painful.

  A rough test: if I described it to a senior engineer over coffee, architecture
   is what I'd draw on a napkin, stack is what I'd list when they asked "ok,
  what'd you build it with?"

  ---
  Your project — architecture

  - Layered MVVM: Composable → ViewModel → Repository (interface) → DataSource 
  (DAO / API). UI knows ViewModel, ViewModel knows Repository, Repository knows
  both DAOs and the API. UI does not know about Room or Retrofit.
  - Unidirectional data flow: ViewModels expose immutable StateFlow<UiState>; UI
   sends events back via method references (viewModel::onEmailChange).
  - Sealed UI state: LocationUiState, SearchUiState (Loading/Success/Error) so
  when is exhaustive.
  - Repository pattern with interfaces: LocationRepository +
  LocationRepositoryImpl, wired by Hilt @Binds. Lets you fake repos in tests.
  - Single source of truth = Room: network refreshes the DB; UI observes the DB
  via Flow. The "cache-then-network" pattern (onStart { refresh() } on a DAO
  flow).
  - String-based navigation via a sealed NavRoutes class; arguments encoded into
   the route path and read in the ViewModel through SavedStateHandle.
  - Stateless JWT auth: token in EncryptedSharedPreferences, attached to every
  outbound request by an OkHttp Interceptor. No session cookies, no per-screen
  auth logic.
  - DI scopes: @Singleton for repos, network client, token store; @HiltViewModel
   per screen.

  Gradle with Kotlin DSL, version catalog (libs.versions.toml), KSP for Room,

  ---
  What about backend, auth model, security?

  Architecture includes cross-cutting decisions like "JWT in Authorization: 
  Bearer, 1h expiry, BCrypt at rest". Those aren't libraries — they're protocol 
  choices. I'd put them under architecture as a "Security model" sub-section,
  the way CLAUDE.md already does.
